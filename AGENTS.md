# AGENTS.md

本文件为仓库级代理协作说明，适用于 Codex 为主、其他代码模型兼容的场景。

## 1. 作用范围

- 本文件作用于仓库根目录及其所有子目录。
- 若子目录存在更近的 `AGENTS.md`，以更近的文件为准。
- 涉及 `backend/` 的实现、测试、接口或重构时，先阅读 `backend/AGENTS.md`。

## 2. 项目概览

- 项目名称：`douyin-parser`
- 目标：提供抖音视频/音频解析下载能力，并提供 AI 对话解析接口。
- 当前核心实现位于 `backend/`，仓库其他目录主要承担文档、流程和辅助信息职责。

## 3. 目录导航

- `backend/`：Node.js 后端服务，项目主要业务代码。
- `docs/`：项目设计与说明文档。
- `README.md`：项目入口说明。
- `.zcf/`、`.spec-workflow/`：过程性文档与计划，不作为运行时代码。
- `.serena/`：工具元数据，不作为业务实现目录。

## 4. 工作原则

- 遵循 `KISS`：优先最直接、最小改动的实现。
- 遵循 `YAGNI`：只实现当前明确需求，不预埋未来特性。
- 遵循 `DRY`：发现重复逻辑时优先抽取到已有边界内。
- 遵循 `SOLID`：保持模块职责单一，依赖方向清晰。

在开始任何非平凡编码任务前，必须先遵循 `karpathy-guidelines` 的约束。即使未显式调用 skill，也应按以下规则执行。

### 1. Think Before Coding
- 不要静默假设需求含义。
- 如果存在歧义、隐含前提或多种合理解释，先明确列出，再说明采用哪一种及原因。
- 如果关键信息不足或上下文冲突，先暂停实现并提出澄清。
- 如果存在更简单的实现路径，应主动指出，不直接沿复杂方案继续。

### 2. Simplicity First
- 只实现当前明确要求的能力。
- 不为单次使用逻辑引入额外抽象。
- 不预埋未来功能、扩展点、配置项或“可能有用”的结构。
- 不为不可能场景补充样板化防御代码。
- 如果实现明显偏重，应主动回退到更直接、更小的方案。

### 3. Surgical Changes
- 只修改与当前任务直接相关的代码。
- 不顺手修改相邻逻辑、注释、格式、命名或无关结构。
- 保持现有代码风格和模块边界一致。
- 只清理由本次改动直接引入的冗余。
- 发现历史遗留问题时，只说明，不擅自清理。

### 4. Goal-Driven Execution
- 先定义成功标准，再开始实现。
- Bug 修复优先复现问题，再修复并验证。
- 增加校验优先补失败场景，再让验证通过。
- 重构必须验证改动前后行为一致。
- 多步骤任务应先给出简短计划，并为每一步指定验证方式。

### 执行要求
- 实现前，先简述假设、边界和计划；简单任务可简化，但不能完全省略。
- 输出结果时，必须说明验证方式；未验证时必须明确说明“未验证”。
- 每一处代码改动都应能直接追溯到用户当前需求。

## 5. 全局协作约束

- 始终先读后写，先理解现有结构与边界，再修改代码。
- 非用户明确要求时，不主动执行 `git commit`、`git push`、建分支、改历史。
- 不修改真实凭据或敏感配置值，如 `backend/.env` 中的密钥。
- 不提交、不编辑 `node_modules/`、运行时缓存、临时产物。
- 不因“顺手”进行无关重构，改动范围应与当前任务直接相关。

## 6. 文档与注释约定

- 输出语言使用简体中文。
- 代码注释与文档优先保持简体中文，与现有仓库一致。
- 修改接口、运行方式或目录职责时，需同步更新相关文档。

## 7. 检索与修改建议

- 需求涉及后端实现时，优先查看 `backend/AGENTS.md`。
- 需求涉及接口说明时，查看 `backend/docs/API.md` 与 `docs/技术方案.md`。
- 需求涉及 AI 能力时，优先从 `backend/ai/`、`backend/routes/ai.js`、`backend/services/AIChatService.js` 开始检索。

## 8. 提交结果期望

- 说明改动落点与原因，而不是罗列无关文件。
- 若未执行测试，应明确说明原因。
- 若发现仓库内已有用户改动且与当前任务冲突，应先暂停并确认。
